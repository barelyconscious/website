I really hate GUI programming. In Script Kitties, I've rewritten the GUI engine twice. The first time was when I went from raw text and boxes to sprites with dynamic sizing:

And the second was when I realized you shouldn't describe GUIs in pure C++.

```cpp
// This is a simple example
Window->AddWidget(RootGUI->NewPanel()
    .Layout(WindowContents)
    .Trim(WidgetTrim::None)
    .AddWidget(RootGUI->NewPanel()
        .Position({ 1, 0, 4, 0 })
        .Size({ 0, 0, 275, 48 })
        .BorderSize(1)
        .BorderColor(FocusedFrameColor)
        .Background({ 0, 0, 0, 255 })
        .AddWidget(RootGUI->NewImage("gui_kittycoin.png")
            .Size({ 0, 0, 48, 48 })
            .AddWidget(RootGUI->NewText()
                .Position({ 1, 0, 5, 18 })
                .Size({ 1, 1, 0, 0 })
                .Update([](WText<ShopState>& Widget, const ShopState& State) {
                    Widget.Text = GameManager::Only().Bag->Money;
                    })
                .Build())
            .Build())
        .Build())
    .Build());
```

### The Developer Experience

Sometimes while working I would take notice of some friction in a process, like constantly rewriting the same lines of code because the logic is needed in multiple places. I can abstract the logic into a function and remove that friction, or I can just keep copying it whenever I need it. If it's only a few lines of code, it may not seem like it's worth it to stop and refactor and sometimes it isn't. How do you know when to refactor? When you haven't been burnt by it, the answer can seem like personal opinion, but really it depends on "how much time will it cost to do it?" and "how much time will it cost to not do it?". The trick is knowing how much time it will take to _not do something_.
