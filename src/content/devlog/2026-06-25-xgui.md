Last week I began writing version 3 of my GUI engine, XGUI (as in *X*ML), because [I hate GUI programming](/devlog/gui-hell). XGUI holds a lot of promise for me because of the visual editing capabilities that fall out organically from the use of XML:

![Editor](/devlog/xgui/wysiwyg.png)

LGUI (as in *L*ua), the previous version, had its entire presentation and controller logic defined in Lua code:
```lua
local GUI = CreateGUI()
local window = GUI:createPanel({ ... })

window.okButton = GUI:createPanel({ ... })
window.okButton.text = GUI:createText({
    text = "Ok"
})

-- Register events
window.okButton:registerEvent("OnMouseClicked", function(mouse) 
    -- handler
end)
```
whereas XGUI, of course, is XML with Lua controllers:
```xml
<View controller="myController.lua">
    <Panel id="window">
        <Panel id="okButton">
            <Text text="Ok" />
        </Panel>
    </Panel>

    <!-- Register events -->
    <Event name="OnMouseClicked" handler="handleOnMouseClicked" />
</View>
```

### Why is this better?

For two reasons: visual editing and strict separation of concerns. Obviously visual editing is going to be a smoother process than editing by hand, but how does having strict separation of concerns help? Two words, four syllables: **spaghetti code**.

![Spaghetti](/devlog/xgui/spaghet.jpg)

Setting clear boundaries between what _this_ code does and what _that_ code does is essential to maintaining a large codebase (and your sanity). Without it, you'll inevitably find yourself fork-deep in a mound of spaghetti code. XGUI follows a simple MVC (Model, View, Controller) pattern. XML defines the View (how the GUI should be displayed) and Lua defines the Controller (how the GUI should behave). The Model is the bridge that the Controller mutates and the View renders.

GUIs are highly interactive, especially in video games. There are mouse events, key events, tooltips, sprites, drag-and-drop, visibility, modals... the list goes on. All the while, there's constant communication between the player and the game engine itself, and everything has to remain in sync. And it takes a _lot_ of code to manage all that. At first, it's easy and convenient to do it all in one place but very quickly you'll find yourself in the middle of a 1,500 line Lua file trying to figure out how to render and control the GUI at the same time you're processing input from the player and handling events from the engine. XGUI - through the MVC pattern - helps _organize_ all of that by forcing the pattern in practice. You simply _can't_ implement View declarations and Controller business logic in the same place.

#### Runtime Bindings

The most common code I wrote in LGUI was updating widgets with data from the game, like a creature's name or sprite for the user to see.

![profile](/devlog/xgui/profile.png)

```lua
-- declare the visual layout
local window = GUI:createPanel({ ... })
window.creaturePanel = GUI:createPanel({ ... })
window.creaturePanel.name = GUI:createText({ ... })
window.creaturePanel.sprite = GUI:createPanel({ ... })
window.creaturePanel.level = GUI:createText({ ... })

-- bind data to those visual elements
function refreshCreatureGUI(creature)
    window.creaturePanel.name:setText(creature.name)
    window.creaturePanel.sprite:setTexture(creature.sprite)
    window.creaturePanel.level:setText("Lv. " .. creature.level)
end

-- later, when the user selects a different creature:
function onCreatureSelected(newCreature)
    refreshCreatureGUI(newCreature)
end
```

Just look at how much code that requires and this is for _every_ dynamic element across _every_ screen. And not only do you need logic to set the values, you also need logic to know when to update those values. This is _everywhere_. It's far more likely that an element needs runtime evaluation rather than a simple static string or sprite. 

It's repetitive, tedious, and as it turns out, completely unnecessary. XGUI's other superpower: _runtime bindings_.

Instead of implementing custom runtime evaluations and update logic, XGUI lets you bind properties (like `text`) to a runtime value (like `name`). No Lua code required.
```xml
<Panel id="creaturePanel">
    <Text id="name" text="{$.name}" />
    <Panel id="sprite" texture="{$.sprite}" />
    <Text id="level" text="{$.level}" />
</Panel>
```

#### Show me the C++

So everything we've talked about so far is great in theory, but how does it _actually work_? Let's get more technical here.

The XML parsing itself is simple enough. I'm using [pugixml](https://pugixml.org/), which seems to be the de facto C++ XML library, to do the actual parsing and I created simple facade objects for a more readable interface.

Widgets are the visual building blocks of XGUI:
```cpp
struct Widget 
{
    string Texture;
    string Text;
    SDL_Color BackgroundColor;
    // other properties...
};
```

The GUILoader is a helper class to do the heavy XML parsing into a C++ handle ("GUI") that the engine knows how to update and render.
```cpp
class GUILoader
{
public:
    GUILoader(const string& Filepath);
    unique_ptr<GUI> Load() const;
};
```

The Widget facade (XWidget) exposes a simple converter to make cosntruction easier. Note the `Bindable` template type.

```cpp
class XWidget
{
public:
    XWidget(pugi::xml_node Node) : Node(Node) {}

    // Handles XML -> Widget conversion
    unique_ptr<Widget> GetWidget() const;

private:
    pugi::xml_node Node;

    // These are all the properties that can appear on an
    // element in the XML doc.
    Bindable<string> Texture() const;
    Bindable<string> Text() const;
    Bindable<SDL_Color> BackgroundColor() const;
    // others...
};
```

A `Bindable` is how we express the nuance between static and runtime values:

```cpp
template <typename T>
class Bindable
{
public:
    string Expression;
    T Value;
};
```

During the `update` call, XGUI evaluates runtime-bound values and updates Widget values so that during the `render` call, the computed values are drawn to the screen.

```cpp
void GUI::RenderNode(const Renderer& Renderer, const Widget& Node) const
{
	if (!Child.Texture.Value.empty())
	{
		Renderer.DrawTexture(
			Child.Texture.Value,
			Bounds.x, Bounds.y, Bounds.w, Bounds.h);
	}
    // other render calls...

	for (auto& ChildNode : Node.GetChildren())
	{
		RenderNode(Renderer, *ChildNode);
	}
}
```

The runtime bindings have proven to be the most fun challenge I've worked on in quite some time. I love implementing ergonomic APIs and interfaces and in general just making code _easy_ to use. This especially is one of those areas where AI just falls flat on its face, but I'll save that rant for another week.

#### So was it worth it?

XGUI came with tradeoffs. It took 2 weeks to make and during that time I wasn't able to work on any of the game's content or core mechanics. XGUI _will_ make it faster and easier to implement GUI screens, but how much time it will save is still theoretical. If the only benefit was faster development, it would be a risky bet.

Something that often gets discounted is the _developer experience_ or DX. When something is easier to use, I've noticed that I'm able to do _more_ with it. When I evolved to LGUI, my screens were able to become much more complex and polished because it was _easier_ to write and reason about. **This** is the real payoff I'm looking for. With MVC, I get clear separation and organization. With the visual editor, I get a simplified interface for designing screens. And with runtime bindings, I avoid 90% of the boilerplate I'm overburdened with. All of that reduces the mental load required to implement GUI screens and allows me to focus better on improving the player experience.

So was it worth it? I think so.
